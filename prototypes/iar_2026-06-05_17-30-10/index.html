Our read

The debugger role says IAR is scaling the part of the platform customers feel every day. IAR Embedded Workbench already joins compiler, linker, analysis, and C-SPY debugging across many architectures. The pressure now is the new path around VS Code, CMake, CI/CD, and container builds. Public comments praise IAR's debugger, but still call the editor old. Some teams switch tools just to edit code. If that handoff stays rough, senior engineers spend time on support loops instead of new trace and RTOS features.

Where we'd start

What we noticed

Separate C-SPY engine work from workflow polish. The public review calls Workbench robust for firmware, yet says the interface feels old.

Useful next step

Map the debugger flows that push users out to VS Code. Turn each gap into a small acceptance test.

What we noticed

Your product pages now sell C-SPY Listwindows, CMake, VS Code, and CI/CD as one path.

Useful next step

Prove those paths on real probes before each release. Keep it small enough to run every week.

What we'd do

Use a dedicated debugger squad under Peter's product and engineering leads. Start with an eight-week pilot, then decide whether to keep the squad for quarterly platform work.
