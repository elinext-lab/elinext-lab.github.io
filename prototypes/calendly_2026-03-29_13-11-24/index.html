The review pattern is specific, not generic

Calendly does not have a weak mobile product. It has a strong mobile product with a repeated failure mode: high-value host actions still break down during exception handling. That is a much more actionable product story than "users want a redesign."

Mobile hosts still hit a desktop escape hatch

Recent reviews ask for blocked-time overrides, mobile reschedules, better availability edits, and meeting polls inside the app instead of forcing users back to web.

App Store + Google Play

Reliability complaints are fresh

March 2026 reviews mention login loops, sign-in failures, slot errors, and stale one-time links. These are high-cost failures because they appear next to real meetings.

App Store + Google Play

Official docs still expose mobile limitations

Calendly's help center says iOS one-off meetings on mobile do not pull connected location data, which supports the broader "not all host tasks are truly mobile-complete" story.

Calendly Help Center

The scale makes small UX leaks expensive

Public company materials cite 20M+ users, 100k+ organizations, and 100+ integrations. At that scale, even modest mobile exception-handling gaps turn into measurable support load.

Calendly fact sheet

What we would pitch the engineering org

Keep the scope additive. This is not a rewrite and not a vanity visual refresh. It is a workflow repair package for the exact moments where the mobile app loses host confidence today.

Offer alternate slots from the same event thread

When a link goes stale or a slot fills unexpectedly, the app should recover instead of forcing the host to rebuild context on desktop.

Override blocked time with team-safe controls

Make exception handling explicit with approvals, reason codes, and audit visibility for shared calendars or sales teams.

Mobile meeting polls on both platforms

This closes an obvious capability gap and gives hosts a simple recovery motion when one slot no longer works.
