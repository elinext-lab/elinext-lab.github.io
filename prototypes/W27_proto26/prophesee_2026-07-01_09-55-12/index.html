Scaling Mantara and Hearth without slowing the core team

Why we're here: On June 15, Prophesee announced Mantara, Hearth, and EUR20M funding. The platform now has to move from expert-led builds to repeatable field delivery.

Our read

The signal says Prophesee is moving from SDK adoption to full system delivery. Mantara is now field-validated, and Hearth becomes the platform after OpenEB and Metavision SDK. That is a good but heavy shift for a CTO team. The hard part is not only AI or sensors. It is migration, release checks, field updates, and support loops that do not slow the research core. If those paths are unclear, pilots can stall before buyers see the real value.

Where we'd start

Observation

SDK setup has many gates

The Linux path needs exact OS, CPU, token, and package steps. We would add a smoke-tested bootstrap for each approved device class.

Next step

Make migration visible

Hearth now replaces OpenEB and the standalone SDK. A compatibility checklist and sample migration path would reduce partner support load.
