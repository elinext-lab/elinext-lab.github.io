The Wedge

HoneyBook does not need another broad dashboard on mobile. It needs one place where AI signals become action before a lead, payment, or project update falls through the cracks.

Unify the signal layer

Pull in Gmail lead finder, follow-up suggestions, priority lists, overdue replies, file blockers, and payment risk so users stop guessing where the next best action lives.

Matches current AI roadmap
Fix the mobile execution moment

Deep-link the rescue action into the exact contract, invoice, file, or message context instead of dropping users into generic screens that trigger more support tickets.

Reduces menu hunting
Close the loop

Track whether the rescue draft was sent, the file blocker was resolved, and the lead recovered. That makes AI value visible to users and measurable to the product team.

Practical Rollout

This can be shipped as a contained product wedge instead of a full mobile rewrite.

Phase 1

Read-only queue that ranks rescue opportunities from existing AI and notification systems. No new intelligence required.

Phase 2

One-tap actions for AI drafts, payment reminders, file requests, and project follow-ups with direct deep-linking.

Phase 3

Recovery analytics that show which actions reopen stalled leads and which mobile blockers create the most support friction.

Built around HoneyBook’s current roadmap, not against it.

This concept keeps HoneyBook’s AI investments intact and makes them easier to trust on mobile. The engineering win is not “more AI.” It is making the existing AI surfaces feel operationally sharp for users who are booking, invoicing, and chasing replies from a phone.
