A stronger candidate path for Totum’s change and law firm searches

We reviewed Totum’s public candidate journey and the live jobs portal. The clearest product is the Firefish-based web portal for search, alerts, registration, and candidate capture.

Our Read

The hiring signal points to a tighter funnel problem

The signal suggests Totum is helping law firms hire for change, not only steady operations. That makes the candidate journey more time-sensitive and more trust-sensitive. Today the path runs across the main site, the Working with Candidates page, live roles, and Firefish login or register steps. If that path feels slow or split on mobile, strong candidates may leave before a consultant speaks with them.

What We’d Do

A short project to fix the public path first

Engagement model: a scoped end-to-end project. Six weeks is enough to map the path, fix the highest-friction points, and ship safe improvements.

Team shape
1 product lead, 1 senior .NET engineer, 1 senior React engineer, 1 QA engineer, 1 UX designer
Relevant stack
.NET, C#, React, Node.js, QA automation

The work stays close to Totum’s current setup. We improve the live candidate path, not replace it with a new product story.

01

Map the live path

Map search, role page, login, register, and apply steps for change and advisory roles. Mark every slow handoff and trust break.

02

Add clear drop-off data

Add light analytics and recruiter views so Totum sees where strong candidates stop, repeat filters, or leave before sending details.

03

Simplify the mobile flow

Simplify mobile filters, role pages, and form states so candidates can act between meetings without losing context or restarting.

04

Make releases safer

Add release checks for routes, cookies, forms, and tracking. This cuts public breakage like dead paths and missed handoffs.
