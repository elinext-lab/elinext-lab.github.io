Our Read

That hire tells us quality is now a growth issue, not a side task. Squishmallows Match runs passes, drops, rewards, and live updates that need clean checks. Public reviews point to missed rewards, ad failures, late fixes, and hard levels. If that keeps leaking, teams spend more time on triage than fresh content.

What We'd Do

Engagement model: we would start with a scoped six-week QA pilot, owned end to end.

Goal: cut live-release risk before it reaches players, support, and partner teams.

1

Map the risk path

Map the risky flows first: event setup, reward grants, purchase paths, ad rewards, and store builds across Squishmallows Match and shared live-service code.

2

Add fast checks

Add fast regression packs for content drops, pass payouts, and recovery cases, so product, QA, and support read the same truth.

3

Build one small tool

Build a small internal check tool for test data, event states, and edge cases, so teams stop proving the same bugs by hand.

4

Leave a working playbook

Run weekly demos with Jim's team, then leave a clear playbook for releases, ownership, and next automation wins.
